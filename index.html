· METHODOLOGY · LOGICS APPLIED TO THE DATA

  • SCOPE   All 7,772 rows in WorkItems. Every row has WorkItemType = Bug — this is a pure bug dataset, not a mixed work-item export.
  • PME IDENTIFICATION   A ticket is considered PME if (a) its title matches the regex PME[\s\-]?\d+, or (b) any of its semicolon-delimited Tags contain "PME". 99.3% of titles match the regex; the remaining 54 are PME-tagged variants.
  • INTERNAL vs EXTERNAL   Derived from Tags: presence of "External PME" → External, "Internal PME" → Internal, else Unclassified. External outnumber Internal ~4:1, confirming this is a customer-facing defect queue.
  • CLIENT EXTRACTION   Tag tokens that are fully uppercase and longer than 5 characters (excluding generic markers like PME, PME_OSM, P1-P5, BLOCKED) are treated as client names. Greystar, ConAm and Camden lead the report volume.
  • PME PRIORITY   Parsed from Tags when a P1–P5 token is present, separately from the workflow Priority field which is the Azure DevOps numeric priority.
  • OPEN vs CLOSED   A ticket is "Closed" if State contains "Closed" (covers "Closed" and "Closed-New Story"). All other states — New, Ready To Work, Active, Resolved, Defined, Test in Progress, Pending More Info, Rework — count as open backlog.
  • RESOLUTION TIME   ClosedDate − CreatedDate in days, computed only for closed tickets. Median is used as the headline metric because the distribution is heavily right-skewed (max > 1,600 days drags the mean).
  • AGE OF OPEN   today() − CreatedDate for tickets still in any open state. The oldest open ticket is 2,300+ days old (2019 vintage), flagging stale backlog.
  • ESCAPE DEFECTS   Computed from the explicit EscapeDefect field where it is filled (Yes / No). Escape rate is the share of "Yes" out of (Yes+No), ignoring the unclassified majority.
  • CUSTOMER-DISCOVERED   DiscoveredBy = "Customer". This is the strongest external-quality signal — >80% of all PME bugs were reported by clients rather than caught internally.
  • FOUR-WAY RCA   The dataset captures four independent RCA dimensions: RCA (high-level), DevRCA, QARCA, and DefectRCA. Each is plotted separately to expose orthogonal patterns — e.g. incorrect logic dominates DevRCA while not identified in impact analysis leads QARCA.
  • PROGRAM HIERARCHY   AreaLevel1 = Business Unit (Consumer Solutions vs FinServ), AreaLevel2 = Program. The full AreaPath backslash-delimited string is preserved in the source data for drill-down.
  • TIME COHORTS   Monthly cohorts pivot on CreatedDate and ClosedDate separately so created-vs-closed crossover reveals whether the team is burning down or accumulating backlog in a given month.
  • HEATMAP   Severity × Program crosstab, top 10 programs by total volume. Cell colour intensity is proportional to count, log-scaled so the Resident Portals cluster doesn't wash out smaller programs.
  • PRIVACY · PII REDACTION   Individual engineer / assignee names have been removed from this dashboard. The original Excel still contains AssignedTo, CreatedBy, ResolvedBy and similar fields for internal use, but they are not visualised here. Corporate client names parsed from Tags are retained as they are entity-level identifiers rather than personal data.
  • DATA HYGIENE   NaN values are kept visible (as "Unspecified") in RCA charts because the absence of an RCA classification is itself an interesting signal — over 50% of tickets have no Dev/QA RCA captured.
